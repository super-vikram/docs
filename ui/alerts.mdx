## Planned Functionality

When implemented, Alerts will provide real-time monitoring of vault health events with severity classification:

| Severity | Examples |
|----------|---------|
| `critical` | PPS staleness exceeded, keeper service down, gas balance critically low |
| `warning` | High redemption queue depth, service success rate below threshold |
| `info` | Scheduled maintenance, parameter changes |

### Planned Alert Lifecycle

```
active → acknowledged → resolved
```

### Planned Integrations

Slack, PagerDuty, Discord, and custom HTTP webhook endpoints.

## Current Workaround

Until the Alerts system ships, monitor vault health using existing endpoints:

1. **Audit logs** — `GET /api/v1/audit/logs` (Erebor, available now)
2. **Service health** — `GET /services/status` (Superman, available now)
3. **Vault state polling** — `GET /api/v1/vaults/{chain_id}/{vault_address}` every 15s for PPS staleness

See the [Monitoring Automation Guide](/automation/monitoring) for production-ready Python and TypeScript polling examples.
