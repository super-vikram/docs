## Four Backend Services

SuperVaults exposes APIs across four separate backend services, each with its own base URL:

| Service | Language | Base URL | OpenAPI Spec |
|---------|----------|----------|-------------|
| **Erebor** | Go | `https://erebor.superform.xyz` | ✅ Available |
| **Superman** | Python | `https://superman-api.superform.xyz` | ❌ Not yet generated |
| **Strategy Engine** | Go | `https://strategy.superform.xyz` | ✅ Available |
| **OMS** | Go | `https://oms.superform.xyz` | ✅ Available |

<Warning>
The **Superman Python** backend (which powers ladder, history, and services) does not yet have an OpenAPI spec. The [Superman API reference](/api/superman) documents its endpoints manually. Treat endpoint shapes as best-effort until a spec is generated.
</Warning>

## Authentication

All endpoints except explicitly public ones require a Bearer JWT:

```
Authorization: Bearer <superman_jwt>
```

The JWT is issued by Dynamic.xyz upon wallet connection. See [Authentication](/authentication) for the full auth flow.

## Common Patterns

### The Prepare Pattern

On-chain operations follow a two-phase flow:

1. **Prepare** — POST to a `prepare-*` endpoint. Returns raw EVM calldata.
2. **Sign** — Pass the calldata to your wallet (`{ to, data, value }`).
3. **Confirm** (where applicable) — Notify Erebor of the tx hash.

Examples: vault creation, yield source add/remove, user management, emergency pause.

```json
// PrepareResult — returned by all prepare-* endpoints
{
  "to": "0xContractAddress...",
  "data": "0xencoded_calldata...",
  "value": "0"
}
```

### Pagination

Erebor endpoints use cursor-based or offset-based pagination:

```
?limit=50&offset=0
?limit=50&cursor=<next_cursor>
```

Most list responses include `has_more` or `next_cursor` for continuation.

### Async Jobs

Long-running operations (vault creation, merkle tree generation) use async job patterns:

1. Trigger the operation → receive a `job_id`
2. Poll `GET /api/v1/.../jobs/{job_id}` with exponential backoff
3. Job status: `pending` → `running` → `completed` / `failed`

### Optimistic Concurrency (Strategy Engine)

Strategy PUT operations include a `version` field. If two clients edit the same strategy simultaneously, the second write returns `409 Conflict`. Refresh and retry with the latest version.

## Error Codes

| Code | Meaning |
|------|---------|
| `400` | Bad request — invalid parameters |
| `401` | Unauthorized — missing or expired JWT |
| `403` | Forbidden — JWT valid but insufficient role |
| `404` | Not found — resource does not exist |
| `409` | Conflict — duplicate or version conflict |
| `500` | Internal server error |

## Rate Limits

Rate limits are not yet published. When integrating automation, implement conservative polling strategies:
- Vault state: poll no faster than every 15 seconds
- Analytics: poll no faster than every 60 seconds
- Services health: poll no faster than every 30 seconds

Implement exponential backoff on `429 Too Many Requests` responses.
