## What is a Curator?

A SuperVaults curator is an on-chain vault operator who manages the allocation of depositor funds across multiple yield sources. Curators configure strategy rules, set keeper parameters, maintain the merkle authorization tree, and monitor operational health.

Curators typically include:
- **DeFi protocols** deploying treasury yield strategies
- **Quant funds** running systematic yield optimization
- **Sophisticated yield managers** offering structured products to depositors

## Responsibilities

### Strategy & Allocation
- Configure deposit priority ladders — which yield sources receive funds first
- Configure redemption priority ladders — which yield sources are unwound first
- Set reserve percentages (assets held in vault vs deployed)
- Define claim reward and swap floor thresholds

### Yield Source Management
- Whitelist yield sources (ERC-4626 vaults, Pendle PTs) via on-chain transactions
- Assign price oracles to each yield source
- Monitor allocation health and rebalance as needed

### Merkle Tree Authorization
- Maintain hook configurations (which on-chain hooks are authorized for which actions)
- Generate and sync merkle roots to chain
- Manage timelock delays for root updates

### Automation
- Define Strategy Engine rules (indicator-based, rule trees)
- Trigger DeFiX intents for deposit, withdrawal, and rebalance actions
- Monitor strategy execution and intent history

### Operations
- Monitor 7 Celery worker services (deposits, redemptions, claims, swaps, queue, cleanup, tx manager)
- Respond to service degradation (pause/resume individual workers)
- Review execution history for deposit and redemption operations

### Access Control
- Manage secondary managers (on-chain delegation)
- Grant/revoke view-only access (off-chain)
- Configure vault settings (redeem timelock)

## Revenue Model

Curators earn fees from two sources:

| Fee Type | Description | Configurable |
|----------|-------------|--------------|
| **Performance Fee** | Basis points applied to yield above high-water mark | Yes (bps) |
| **Management Fee** | Annualized basis points on TVL | Yes (bps) |

Both fees are split between the curator (as recipient) and the Superform protocol (via SuperGovernor). The revenue share split is defined in the SuperGovernor config per chain.

<Note>
Fee configuration happens at vault creation time. The `SuperGovernorConfig` defines protocol-level constraints including minimum staleness, upkeep payment requirements, and protocol revenue share.
</Note>

## Vault Lifecycle

```
CREATE    →    CONFIGURE    →    ENABLE    →    OPERATE    →    (PAUSE / SHUTDOWN)
```

| Phase | Actions |
|-------|---------|
| **Create** | Deploy vault (choose chain, asset, name, fees). Sign on-chain tx. |
| **Configure** | Whitelist yield sources. Build hook config. Generate merkle tree. Sync root to chain. |
| **Enable** | Toggle vault enabled in Superman. Verify services are healthy. |
| **Operate** | Monitor PPS, TVL, keeper health. Adjust ladders. Run strategies. |
| **Pause** | Emergency on-chain pause via Erebor prepare pattern. Or pause individual keepers. |

## Permission Levels

| Role | On-Chain? | Capabilities |
|------|-----------|--------------|
| **Primary Manager** | ✅ Yes | Full control. Set at vault creation. Immutable. |
| **Secondary Manager** | ✅ Yes | All vault operations except vault settings. Added via on-chain tx. |
| **View Only** | ❌ No | Dashboard read access only. Off-chain DB record. |
| **Registry Maintainer** | ❌ No | Global registry CRUD (yield sources, oracles, hooks). Separate permission. |

<Tip>
Use secondary managers for team operations and automation accounts. Reserve the primary manager key for governance-level decisions only.
</Tip>
