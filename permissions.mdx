## Role Overview

SuperVaults has four permission tiers. Each tier has a different on-chain vs off-chain footprint:

| Role | On-Chain? | Scope | Mutable? |
|------|-----------|-------|---------|
| **Primary Manager** | ✅ Yes | Full vault control | Immutable (set at creation) |
| **Secondary Manager** | ✅ Yes | All operations except vault settings | Removable via on-chain tx |
| **View Only** | ❌ No | Dashboard read only | Added/removed off-chain |
| **Registry Maintainer** | ❌ No | Global registry CRUD | DB-level grant |

## Primary Manager

The primary manager is the on-chain vault owner. There is exactly one primary manager per vault.

- Set at vault creation time — the wallet that signs `POST /api/v1/vaults/create/prepare`
- Cannot be changed without governance-level intervention
- Has exclusive access to vault settings (redeem timelock)
- Can add and remove secondary managers

<Warning>
Guard your primary manager key carefully. It cannot be rotated without on-chain contract interaction. Use a hardware wallet or multi-sig for primary manager operations in production.
</Warning>

## Secondary Manager

Secondary managers are on-chain delegates. A vault can have multiple secondary managers.

- Added via `POST /api/v1/users/secondary-manager/prepare` (prepare pattern — requires wallet signature)
- Removed via `POST /api/v1/users/secondary-manager/remove/prepare`
- Can perform all vault operations: ladder changes, yield source management, merkle tree operations, strategy control, pause/unpause
- Cannot modify vault settings (primary manager exclusive)

## View Only

View-only access is entirely off-chain (no on-chain transaction required).

- Added via `POST /api/v1/users/view-only` (JWT-authenticated POST, no wallet sig)
- Removed via `DELETE /api/v1/users/view-only/{wallet}`
- Access grants read-only dashboard visibility
- No ability to modify vault configuration

## Registry Maintainer

A separate permission tier for maintaining the global Erebor registries.

- Grants access to the **Registry** page and **Admin** page
- Not vault-specific — applies globally
- Required for: creating/updating/deleting yield sources, oracles, hooks; running data backfills

## Permission Matrix

| Action | Primary | Secondary | View Only | Registry Maintainer |
|--------|---------|-----------|-----------|---------------------|
| View dashboard | ✅ | ✅ | ✅ | — |
| Configure ladders | ✅ | ✅ | ❌ | — |
| Manage yield sources | ✅ | ✅ | ❌ | — |
| Merkle tree management | ✅ | ✅ | ❌ | — |
| Strategy creation/edit | ✅ | ✅ | ❌ | — |
| Pause vault (on-chain) | ✅ | ✅ | ❌ | — |
| Pause keeper services | ✅ | ✅ | ❌ | — |
| Add secondary manager | ✅ | ❌ | ❌ | — |
| Remove secondary manager | ✅ | ❌ | ❌ | — |
| Add/remove view-only | ✅ | ✅ | ❌ | — |
| Vault settings (timelock) | ✅ | ❌ | ❌ | — |
| Registry CRUD | — | — | — | ✅ |
| Admin backfill | — | — | — | ✅ |

## AuthGuard Enforcement

The Curator App enforces permissions at the page level via `AuthGuard`:

- `requireManager` — checks `superman_is_manager` in localStorage; redirects to login if false
- `requirePrimaryManager` — checks vault-specific primary role
- `requireRegistryMaintainer` — checks registry maintainer flag

All Erebor API endpoints also enforce permissions server-side. A `403 Forbidden` response means the connected wallet lacks the required role for that operation.

## Managing Users

See [User Management](/ui/user-management) for the step-by-step UI guide, or use the Erebor API directly:

```bash
# List all users for a vault
GET /api/v1/users?vault_address={address}&chain_id={chainId}
Authorization: Bearer <jwt>

# Add a secondary manager (returns calldata to sign)
POST /api/v1/users/secondary-manager/prepare
Authorization: Bearer <jwt>
Content-Type: application/json

{
  "vault_address": "0xdef...",
  "chain_id": 8453,
  "manager_address": "0xnew..."
}
```
